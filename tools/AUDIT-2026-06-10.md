## Findings

### A. Agent personalities (all 10 .agent.md read in full)

Roster: search, quality-reviewer, source-verifier, contradiction-detector,
challenge, connect, emerge, idea-discovery, cos, enhancer, ingest (11 files;
wiki-agent.py is the orchestrator, not a personality).

**A1. Clean orthogonal core (no action).** The audit trio is well-factored on a
single axis: quality-reviewer = ONE page intrinsic; source-verifier = ONE page
vs raw/sources-text; contradiction-detector = N pages vs each other. Scope
"Owns / Does NOT do" blocks mutually reference each other consistently.
enhancer vs ingest split (first-pass vs re-pass on an already-listed source) is
also clean and stated identically on both sides.

**A2. "What should I do next?" is answered by THREE agents — partial overlap.**
- wiki-emerge: unnamed patterns from recent activity, each with "Suggested action".
- wiki-idea-discovery: ranked shortlist from explicit loose ends (inventory,
  TODOs, orphans, sparse pages).
- wiki-cos (brief mode): "Today's focus" + mandatory "Recommended next action".
The conceptual boundaries (implicit pattern / explicit backlog / deadline-driven
task triage) are defensible, but all three scan the same inputs (wiki/log.md
tail, projects/*/TODO.md, inventory) and all three emit a next-action
recommendation. Concretely non-orthogonal bits:
  - cos "Neglected" section duplicates emerge's "repeated blockers" detection
    (both grep recent log entries for missing project slugs).
  - emerge's per-pattern "Suggested action" encroaches on idea-discovery's
    ranking job; emerge's handoff even says to send through-lines to itself
    backwards (discovery -> emerge -> discovery loop possible).
Suggested fix: make emerge purely descriptive (drop "Suggested action", keep
handoff to idea-discovery), and have cos's Neglected section defer to emerge
(or share one definition of "neglect": N open items + absent from last 15 log
entries — currently only cos has the quantified rule, emerge has none).

**A3. Duplicated boilerplate = maintenance redundancy, not personality overlap.**
The identical "Pre-approved shell commands" paragraph (same 16-command list +
same enforcement note) is copy-pasted in 8 of 11 files; "Citation discipline"
and the `[outside wiki — agent inference]` marker are copy-pasted in the 4
thinking agents. Changing the allowlist means editing 8 files in lockstep
(cos already drifted: it embeds the list in prose with a different shape and
adds an explicit deny-list nobody else has). The orchestrator (wiki-agent.py)
reportedly builds the per-command allowlist anyway — single-source it there or
in a shared include, and keep only agent-specific deltas in each file.

### B. wiki-agent.py bugs (read in full, lines cited)

**B1. [BUG, high] Claude `--allowedTools` Bash pattern likely wrong syntax.**
`wiki-agent.py:772-774` builds `Bash({c} *)` (space-star). Claude Code's
documented prefix rule is `Bash(cmd:*)` (colon-star); `Bash(grep *)` is treated
as an exact match on the literal string `grep *`, which never matches a real
command. Consequence: under `--cli claude` (the DEFAULT cli, and the scheduler
backend since 2026-06-02) every shell call by a shell-enabled agent is NOT
pre-approved; in headless `-p` mode unapproved tools are denied. Read-only
agents would limp along Read-only; enhance/ingest lose grep/sed/awk/wiki.py.
`[unverified — confirm in 2 min: `claude -p --allowedTools "Bash(echo *)" 
"run: echo hi"` vs `"Bash(echo:*)"` and see which executes without asking]`.
Fix: `f"Bash({c}:*)"`.

**B2. [BUG, medium] Search query is attached as a file path.**
`wiki-agent.py:957-961`: for `search`, extra_args=[query] whenever non-empty —
the code comment says "only attach a file when a concrete path was given" but
there is no path check. The free-text query then becomes `-f "<query>"` under
opencode (bogus file) and a `Files to read:\n- <query>` block under
claude/copilot (model wastes a turn failing to Read it). Fix: only attach when
`(ROOT / query).exists()`.

**B3. [BUG, medium] Ingest is double-logged.**
wiki-ingest.agent.md Maintenance says "Append log entry" (agent writes one via
append-log inside the sandbox — wiki/ is writable for ingest), AND the launcher
unconditionally calls `_auto_log_ingest` after rc==0 (`wiki-agent.py:1006-1010`).
Every successful ingest gets two wiki/log.md entries (the launcher one is
low-information: "Ingested X via wiki-agent.py"). Pick one: drop the launcher
auto-log, or tell the agent personality not to log (launcher owns it).

**B4. [dead code, low] Codex effort auto-upgrade can never fire in practice.**
`wiki-agent.py:944-946` upgrades effort only when `effort == "medium"`, but the
parser default is `"high"` (line 499). Only a user explicitly passing
`--effort medium` with a codex model triggers it — contradicts the comment
"deep reasoning by default".

**B5. [quirk, low] EFFORT_MAP: opencode "high" and "xhigh" are identical**
(both `--variant xhigh`, lines 367-378) — if intentional, comment it.

**B6. [inconsistency, low] Path resolution asymmetry**: quality/verify/enhance
attach `(ROOT/path).resolve()` absolutes; ingest's `_resolve_pdf_to_markdown`
returns the ORIGINAL relative string for non-PDF sources (line 56-57) —
opencode `-f` then depends on cwd.

**B7. [gap, low] Non-shell agents (quality, verify) get only `Read`** in the
claude branch — no Grep/Glob tools (line 770). Ties to A4: quality-reviewer's
own criteria (link validity, orphan detection) are un-executable with Read
alone. Either add Grep/Glob to allowed_tools for bash:false agents, or trim
the personality's criteria.
